First-time setup for course-file downloads. Runs **once** per workspace, the
first time the user asks to download/sync course material. After init the two
generated markdown files are the source of truth — the Pre-Download
Confirmation prompt in `file-downloads.md` reads them instead of asking the same
3 questions every time.

## When to run init

At the start of any **download file / sync course material** task:

1. Compute the workspace root = the directory the user is currently working in
   (the repo root they invoked the skill from).
2. Check whether `SYNC_PREFERENCES.md` and `USER_NOTED_FILES.md` exist there.
3. If **both exist** → skip init, go straight to the normal download flow.
4. If **either is missing** → run init now, then continue to the download flow.

init is **not** triggered by deadline/grade/announcement tasks — only by
downloads. Don't run it "just in case".

## Init dialogue (4 questions, scan-then-ask)

The agent does **not** ask blank questions. Run `scan_workspace.py` first, then
use its JSON output to pre-fill each question. Present one question at a time
via the `question` tool, in this order:

### Step 0 — scan

```bash
python3 scripts/scan_workspace.py        # scans cwd
# or:
python3 scripts/scan_workspace.py /explicit/path
```

Keep the JSON in context — every following question reuses it.

### Q1 — Destination (default: user's current folder)

The scanned JSON's `destination_suggestion` is the workspace root itself
(`"默认的下载地址是用户当前打开的folder"`). If `existing_download_dirs` is
non-empty, surface those as alternatives in the options.

Ask:
> Where should Brightspace files be downloaded to?
> - (Recommended) `<workspace_root>`  *(your current folder)*
> - `<workspace_root>/downloads`  *(if existing_download_dirs includes it)*
> - Custom path — type your own

Write the chosen path into `answers.destination`.

### Q2 — Folder layout

Use `layout_suggestion` from the scan. If the workspace already has a course or
type folder structure (`existing_structure_hint`), match it; otherwise default to
`course-first`.

Ask:
> How should files be organized inside the download root?
> - (Recommended) course-first — `ECE327/lectures/file.pdf`  *(matches your current structure / default)*
> - type-first — `lectures/ECE327/file.pdf`
> - Custom — describe your own scheme

Write `course-first` or `type-first` (or a free-text custom scheme) into
`answers.layout`.

### Q3 — Auto-detected files to protect (checklist)

From the scan's `potential_protected_files`, present the list as a
multi-select so the user can tick which ones must never be overwritten. Also show
the default `protected_patterns_default` and let the user keep/remove them.

Ask (multi-select):
> I scanned your workspace and found these files that look like personal notes
> or annotations. Which ones should I **never** overwrite?
> 
> Patterns (auto-protect anything matching):
>   [x] my-notes-*.md
>   [x] annotated-*.md
>   [x] annotated-*.pdf
>   [x] *-personal.*
> 
> Specific files found:
>   [ ] SKILL.md                          (markdown)
>   [ ] notes/lec-03-mine.md             (name matches personal pattern)
>   [ ] ECE327/annotated-lab-01.pdf      (name matches personal pattern)
>   ...

Collect:
- kept patterns → `answers.protected_patterns`
- checked files → add to `answers.protected_explicit`

### Q4 — Anything else to protect? (free text)

> Any other files or folders I should never overwrite? Paste paths or globs,
> or hit Enter to skip.

Append user's answer (split by newlines) to `answers.protected_explicit`.

**(Optional Q5)** If the user mentioned a specific course mapping during the
dialogue (e.g. "ECE327 should go under ECE327/Lectures/"), capture it as
free text into `answers.course_mapping`. Skip otherwise.

## Writing the files

Pipe the assembled `answers` JSON into `write_sync_config.py` via stdin:

```bash
echo '<answers JSON>' | python3 scripts/write_sync_config.py
```

The script:
- writes `<workspace_root>/SYNC_PREFERENCES.md`
- writes `<workspace_root>/USER_NOTED_FILES.md`
- backs up any pre-existing versions with a `.YYYYMMDD-HHMMSS.bak` suffix
- prints a JSON summary of written paths + backups

After writing, tell the user both file paths and that they can edit them by hand
anytime — init won't run again as long as both files exist.

## `SYNC_PREFERENCES.md` format

```markdown
# Sync Preferences
<!-- Generated by brightspace-student-hub init — editable by hand. -->

## destination
/Users/you/your-repo

## layout
course-first

## course_mapping  (optional)
<!-- Override default folder derivation by course. -->
```
ECE327 -> ECE327/Lectures
ECE350 -> ECE350
```

## conflict_policy
skip  # default — preserve existing files. change to overwrite | rename if desired.
```

- `destination` / `layout` / `conflict_policy` are the three knobs the
  Pre-Download Confirmation prompt used to ask every time. Now read from here.
- `course_mapping` is optional free text; when present, it OVERRIDES the
  default `course_to_folder()` derivation in `file-downloads.md`.

## `USER_NOTED_FILES.md` format

```markdown
# Protected Files (never overwritten, regardless of conflict policy)
<!-- Generated by brightspace-student-hub init — maintain by hand. -->

## patterns (auto-protect)
- my-notes-*.md
- annotated-*.md
- annotated-*.pdf
- *-personal.*

## explicit files
- ECE327/Lectures/lecture-03-annotated.pdf
- notes/scratch.md
```

Two tracks:
- `patterns` — glob-style, protects any matching file automatically
- `explicit` — individual relative paths

Both are honored during every download — even if the user picks "overwrite" as
the conflict policy, these files are still skipped. This is the user's hard
override of the conflict policy.

## Re-running init

If the user asks to change sync settings later (new destination, protect more
files), they can either:
- edit the two md files by hand, or
- ask the agent to "re-run sync init" — the agent runs scan + dialogue again and
  `write_sync_config.py` backs up the old versions before overwriting.
